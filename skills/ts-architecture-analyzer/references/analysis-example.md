## Execution Walkthrough

### findById
1. Receives domain identifier string
2. Delegates to TypeORM `findOne({ where: { id } })`
3. Returns `null` if not found (caller responsible for 404)
4. Maps entity to domain model via `InventoryMapperPostgres`

### findAll
1. Builds TypeORM `FindOptionsWhere` from filter DTO
2. Executes `find()` with optional pagination
3. Maps entity array to domain model array

**Design decisions:**
1. **Null return on not-found** — Probable reason: keep domain layer responsible for business errors
2. **Mapper delegation** — Probable reason: single responsibility, avoid fat repository

## Risk Analysis

### Gotchas
| Symptom | Cause | Triggering Condition |
|---------|-------|---------------------|
| Silent data loss on save | No transaction wraps related inserts | Concurrent requests modifying same inventory item |
| Untyped DB error in response | `QueryFailedError` not caught and mapped | PostgreSQL constraint violation or connection drop |
| Stale data returned | No cache invalidation | Multiple instances behind load balancer with read replicas |

### Failure Points
| Scenario | Degradation | Threshold |
|-----------|-------------|--------|
| 500+ concurrent findAll calls | Connection pool exhaustion | TypeORM default pool size = 10 |
| Large filter result without pagination | Memory spike + slow response | >10,000 rows returned at once |
| DB replica lag > 2s | findById returns stale or missing data | Replica sync delay under write load |

### Pro-Tips
| Problem | Solution | Trade-off |
|----------|----------|-----------|
| Untyped DB errors | Catch `QueryFailedError` and map to domain `RepositoryException` | Requires maintaining error mapping |
| No transaction management | Use `DataSource.transaction()` for multi-step writes | Adds complexity to caller |
| Missing pagination | Add `take`/`skip` to all `find()` calls | Requires API contract change |
