# python-test-governor

## Description

Audits unit test coverage in a Python repository. Maps every production module to its test status,
classifies gaps by severity (CRITICAL / HIGH / MEDIUM / LOW), determines the correct action per module
(create-test / update-test / no-change / no-test-needed), and outputs a structured governance report
with prioritized recommendations. Optionally integrates with coverage.py XML output and git diffs
to focus the analysis on changed code.

## When to Use

Invoke this skill when you need to:

- Review unit test coverage state of a Python repository before a release or sprint review
- Understand which modules lack tests and why they matter
- Make coverage decisions tied to changed code in a PR or branch
- Generate a structured report for the team or for CI gating
- Determine whether to create new tests or update existing ones for a given module

Use this skill before `/tw-python-test-remediator` — the governor produces the report,
the remediator turns it into code changes.

## Instructions

Follow these steps **in order**, every time. Do not skip steps or reorder them.

### Step 1 — Run Tests and Generate Inventory

**Always start here.** Run tests first to capture current state, then run the inventory script:

```bash
# Run tests and generate coverage (continue even if tests fail)
pytest --cov=. --cov-report=xml --cov-report=html || true

# Generate comprehensive inventory
bash .ai/toolkit/skills/python-test-governor/scripts/repo_test_inventory.sh \
  --base-ref origin/main \
  --coverage-xml coverage.xml \
  --test-results-dir test-results
```

Omit `--base-ref` if not analyzing a diff. Omit `--coverage-xml` if coverage.py was not run.

The script outputs these sections:
- `TEST INVENTORY` — all test files classified as `unit`, `integration`, or `support`
- `MODULE TO TEST MAP` — production modules mapped to tests (`covered` or `unmapped`)
- `PRIORITY GAPS` — logic-heavy modules without any test file
- `CHANGED MODULES` — modules modified in diff (only when `--base-ref` provided)
- `CHANGE DECISIONS` — recommended action per changed module
- `FAILING TESTS` — tests that failed, with error summary
- `TEST MAINTENANCE DECISIONS` — tests needing update or review

### Step 2 — Analyze Inventory Output

Parse the script output to categorize:

1. **Failing tests** — check if systemic (same error pattern) or individual
2. **Unmapped modules** — classify by coverage priority (see `references/analysis-guide.md`)
3. **Changed modules** — inspect diff to determine if tests need update
4. **Covered modules** — verify tests are passing and adequate

### Step 3 — Inspect Representative Code

Do not analyze in isolation. Read actual source files:

- Read 2–3 production modules (especially failing test targets or high-priority gaps)
- Read 2–3 test modules to understand existing patterns and quality
- Read configuration files (`setup.py`, `pyproject.toml`, `pytest.ini`)
- Read test output files when tests are failing

### Step 4 — Classify Severity and Decisions

For each finding, apply the rules in `references/analysis-guide.md`:

- **Severity**: CRITICAL → HIGH → MEDIUM → LOW
- **Decision**: `create-test` / `update-test` / `no-change` / `no-test-needed`

Never use vague language. Use numbers: "39 tests failing", not "many tests failing".

### Step 5 — Generate Structured Report

Use the template in `references/output-template.md`. The report has 7 required sections:

1. **Executive Summary** — repository analyzed, overall decision, primary risk
2. **Findings** — markdown table: Area | File | Finding | Severity | Action (minimum 3 rows)
3. **Impacted Modules** — markdown table: every production module with test status and decision
4. **Root Cause Analysis** — only when 5+ tests fail with the same error pattern
5. **Suggested or Generated Tests** — markdown table with Status: BLOCKED / READY / GENERATED
6. **Risks and Limitations** — minimum 3 concrete risks in bullet format
7. **Recommendations** — Priority 1 (CRITICAL) / Priority 2 (HIGH) / Priority 3 (MEDIUM)

**Always end recommendations with:**
"Would you like me to: 1) [option], 2) [option], 3) [option]?"

Overall decision values:
- `HALT` — critical blockers prevent any test work
- `FIX TESTS FIRST` — existing tests broken, fix before creating new ones
- `CREATE TESTS` — no blockers, proceed with new test creation
- `UPDATE TESTS` — focus on updating existing tests
- `NO ACTION NEEDED` — coverage is adequate

### Step 6 — Generate or Suggest Tests (if requested)

Only after the structured report is complete. Follow standards in `references/test-generation-standards.md`:
- Match project framework (pytest or unittest)
- Arrange / Act / Assert pattern
- One behavior per test
- Deterministic inputs, specific assertions
- Mock external dependencies, not the code under test

### Step 7 — Save Report to File

After presenting the report in the conversation, always save it:

```bash
filename="TEST_GOVERNANCE_REPORT_$(date +%Y-%m-%d_%H-%M-%S).md"
```

Include at the top of the file:
```markdown
# Python Unit Test Governance Report

**Generated**: YYYY-MM-DD HH:MM:SS
**Skill**: /tw-python-test-governor
**Skill version**: v0.1.0
```

Save to the project root using the Write tool.

### Classification Rules

**Unit test** — most of these are true:
- Lives under `tests/` or named `test_*.py` / `*_test.py`
- Uses pytest, unittest, or nose
- Instantiates the module/class directly or with mocks/patches
- Does not depend on DB, HTTP, filesystem, or external services
- Executes in < 1 second per test

**Integration test** — uses any of:
- SQLAlchemy, Django ORM, or database connections
- HTTP clients (requests, httpx)
- Actual file I/O
- External services (Redis, queues)
- Docker or testcontainers

Out of scope: `support` files (conftest.py, fixtures, helpers) and integration tests are excluded
from gap analysis and remediation unless the user explicitly expands the task.

### Coverage Priority Order

Prioritize gaps in this order:
1. Business logic, services, utilities, validators, transformers
2. Modules with conditionals, error handling, data processing, or API interaction
3. Public functions with branches, exceptions, null handling, or complex logic
4. Regressions revealed by changed code or failed tests

Do NOT prioritize: simple dataclasses, configuration constants, empty `__init__.py`,
simple exception classes, TypedDict / NamedTuple definitions.

### Decision Rules for Changed Code

**Create new tests when:**
- A new module with business logic is added and no matching test exists
- A public function is introduced without equivalent assertions
- Error handling, fallback, or transformation logic is added

**Update existing tests when:**
- The module has a test but the diff changes behavior, contracts, branches, or dependencies
- A function signature, exception type, or conditional path changes
- Existing assertions are now too weak, brittle, or obsolete
- Test output shows failures related to changed code

**Do not change tests when:**
- The diff is limited to formatting, renaming, logging, comments, or docstrings
- The module is non-executable plumbing with no meaningful logic change

When uncertain, prefer a small targeted test over no test.

### CI and PR Gating

When the user asks for CI gating configuration, use `references/ci-gating-rules.md`.
Recommend a layered approach:
1. Fast pre-commit hook on changed Python files
2. PR validation with unit tests, coverage.xml, and diff-aware governance
3. Optional nightly deep analysis for stale tests and coverage drift

## Examples

### Example 1 — Full audit of a Python repo

```
/tw-python-test-governor
```

Claude will run the inventory script, analyze all production modules, and produce a full
governance report with findings, module map, and prioritized recommendations.

### Example 2 — Diff-focused audit on a PR branch

```
/tw-python-test-governor base_ref=origin/main coverage_xml=coverage.xml
```

Claude will focus the analysis on modules changed in the current branch, classify whether
each changed module needs new or updated tests, and produce a targeted report.

### Example 3 — Full audit with test result integration

```
/tw-python-test-governor coverage_xml=coverage.xml test_results_dir=test-results
```

Claude will identify failing tests, diagnose systemic issues, and produce a report
that includes root cause analysis when 5+ tests share the same error pattern.

### Example output structure

```markdown
## Executive Summary

- **Repository analyzed**: /path/to/my-service
- **Overall decision**: CREATE TESTS
- **Primary risk**: Core data processing module has zero unit coverage

## Findings

| Area | File | Finding | Severity | Action |
|------|------|---------|----------|--------|
| Coverage Gap | `data_processor.py` | Business logic with 6 functions — unmapped | HIGH | create-test |
| Coverage Gap | `validator.py` | Input validation logic — no test file | HIGH | create-test |
| Unit Test | `test_utils.py` | 2 tests failing — AttributeError | MEDIUM | update-test |
```
