# ts-test-governor

## Description

Audits Jest unit test coverage in a TypeScript/NestJS repository. Maps every production module
to its spec file status, classifies gaps by severity (CRITICAL / HIGH / MEDIUM / LOW), determines
the correct action per module (create-test / update-test / no-change / no-test-needed), and outputs
a structured governance report with prioritized recommendations. Optionally integrates with
`jest --coverage` JSON summary and git diffs to focus the analysis on changed code.

## When to Use

Invoke this skill when you need to:

- Review Jest unit test coverage state of a TypeScript/NestJS repository before a release or sprint review
- Understand which modules lack spec files and why they matter
- Make coverage decisions tied to changed code in a PR or branch
- Generate a structured report for the team or CI gating
- Determine whether to create new tests or update existing ones for a given module

Use this skill before `/tw-ts-test-remediator` — the governor produces the report,
the remediator turns it into code changes.

## Instructions

Follow these steps **in order**, every time. Do not skip steps or reorder them.

### Step 1 — Run Tests and Generate Inventory

**Always start here.** Run tests first to capture current state, then run the inventory script:

```bash
# Run tests and generate coverage (continue even if tests fail)
npm run test:cov -- --coverage-reporters=json-summary || true

# Generate comprehensive inventory
bash .ai/toolkit/skills/ts-test-governor/scripts/repo_test_inventory.sh \
  --base-ref origin/master \
  --coverage-summary coverage/coverage-summary.json \
  --test-results-dir test-results
```

Omit `--base-ref` if not analyzing a diff. Omit `--coverage-summary` if jest --coverage was not run.

The script outputs these sections:
- `TEST INVENTORY` — all spec files classified as `unit`, `integration`, or `support`
- `MODULE TO TEST MAP` — production modules mapped to spec files (`covered` or `unmapped`)
- `PRIORITY GAPS` — logic-heavy modules without any spec file
- `CHANGED MODULES` — modules modified in diff (only when `--base-ref` provided)
- `CHANGE DECISIONS` — recommended action per changed module
- `FAILING TESTS` — tests that failed, with error summary
- `TEST MAINTENANCE DECISIONS` — tests needing update or review

### Step 2 — Analyze Inventory Output

Parse the script output to categorize:

1. **Failing tests** — check if systemic (same error pattern) or individual
2. **Unmapped modules** — classify by coverage priority (see Coverage Gap Rules)
3. **Changed modules** — inspect diff to determine if tests need update
4. **Covered modules** — verify tests are passing and adequate

### Step 3 — Inspect Representative Code

Do not analyze in isolation. Read actual source files:

- Read 2–3 production modules (especially failing test targets or high-priority gaps)
- Read 2–3 spec files to understand existing patterns and quality
- Read configuration files (`package.json`, `jest.config.ts`, `tsconfig.json`)
- Read test output files when tests are failing

### Step 4 — Classify Severity and Decisions

For each finding, apply the rules in `references/analysis-guide.md`:

- **Severity**: CRITICAL → HIGH → MEDIUM → LOW
- **Decision**: `create-test` / `update-test` / `no-change` / `no-test-needed`

Never use vague language. Use numbers: "12 spec files failing", not "many tests failing".

### Step 5 — Generate Structured Report

Use the template in `references/output-template.md`. The report has 7 required sections:

1. **Executive Summary** — repository analyzed, overall decision, primary risk
2. **Findings** — markdown table: Area | File | Finding | Severity | Action (minimum 3 rows)
3. **Impacted Modules** — markdown table: every production module with test status and decision
4. **Root Cause Analysis** — only when 5+ tests fail with the same error pattern
5. **Suggested or Generated Tests** — markdown table with Status: BLOCKED / READY / GENERATED
6. **Risks and Limitations** — minimum 3 concrete risks in bullet format
7. **Recommendations** — Priority 1 (CRITICAL) / Priority 2 (HIGH) / Priority 3 (MEDIUM)

**Always end recommendations with:**
"Would you like me to: 1) [option], 2) [option], 3) [option]?"

Overall decision values:
- `HALT` — critical blockers prevent any test work
- `FIX TESTS FIRST` — existing specs broken, fix before creating new ones
- `CREATE TESTS` — no blockers, proceed with new spec creation
- `UPDATE TESTS` — focus on updating existing specs
- `NO ACTION NEEDED` — coverage is adequate

### Step 6 — Generate or Suggest Tests (if requested)

Only after the structured report. Follow standards in `references/test-generation-standards.md`:
- Jest with `@nestjs/testing` (match project conventions)
- `jest.fn()`, `jest.spyOn()`, `jest.mock()` for mocking
- Arrange / Act / Assert pattern
- One behavior per test
- `beforeEach` / `afterEach` for setup/teardown
- `it.each` / `describe.each` for parameterized cases

### Step 7 — Save Report to File

After presenting the report, always save it:

```bash
filename="TEST_GOVERNANCE_REPORT_$(date +%Y-%m-%d_%H-%M-%S).md"
```

Include at the top:
```markdown
# TypeScript Unit Test Governance Report

**Generated**: YYYY-MM-DD HH:MM:SS
**Skill**: /tw-ts-test-governor
**Skill version**: v0.1.0
```

Save to the project root using the Write tool.

### Classification Rules

**Unit spec** — most of these are true:
- File has `*.spec.ts` naming, colocated with the module or under `__tests__/`
- Uses Jest and optionally `@nestjs/testing` `TestingModule`
- Instantiates the class directly or with `jest.fn()` mocks for all dependencies
- Does not depend on real databases, HTTP calls, Kafka, or filesystem I/O
- Executes in < 1 second per test

**Integration test** — uses any of:
- TypeORM real connections (`DataSource`, `Repository` with real DB)
- Real HTTP calls (`axios`, `supertest` against running app)
- Real Kafka/messaging connections
- Filesystem operations (actual file I/O)
- Docker or testcontainers

Out of scope: `support` files (shared fixtures, test helpers) and integration specs are excluded
from gap analysis and remediation unless the user explicitly expands the task.

### Coverage Priority Order

Prioritize gaps in this order:
1. Business logic services, use cases, domain entities, validators, mappers, transformers
2. Modules with conditionals, error handling, data processing, or API interaction
3. Public methods/functions with branches, exceptions, null handling, or complex logic
4. Regressions revealed by changed code or failed tests

Do NOT prioritize: `*.module.ts`, simple TypeORM entities with only column decorators (`*.entity.ts`),
DTO/interface-only files (`*.dto.ts`, `*.type.ts`, `*.enum.ts`), barrel files (`index.ts`),
entry point (`main.ts`), simple exception classes.

### Decision Rules for Changed Code

**Create new tests when:**
- A new module with business logic is added and no matching `*.spec.ts` exists
- A public method or function is introduced without equivalent assertions
- Error handling, fallback, or transformation logic is added

**Update existing tests when:**
- The module has a spec but the diff changes behavior, contracts, branches, or dependencies
- A method signature, exception type, or conditional path changes
- Existing assertions are now too weak, brittle, or obsolete
- Test output shows failures related to changed code

**Do not change tests when:**
- The diff is limited to formatting, renaming, logging, comments, or type annotations
- The module is non-executable plumbing with no meaningful logic change

### CI and PR Gating

When the user asks for CI gating, use `references/ci-gating-rules.md`.
Recommend a layered approach:
1. Fast pre-commit hook on changed TypeScript files (husky + lint-staged)
2. PR validation with unit tests, coverage-summary.json, ESLint/tsc, and diff-aware governance
3. Optional nightly deep analysis for stale tests and coverage drift

```

```

```

```


### Example output structure

```markdown
## Executive Summary
- **Repository analyzed**: /path/to/my-nestjs-service
- **Overall decision**: CREATE TESTS
- **Primary risk**: Core order service has zero unit test coverage

## Findings
| Area | File | Finding | Severity | Action |
|------|------|---------|----------|--------|
| Coverage Gap | `orders.service.ts` | Business logic with 8 methods — unmapped | HIGH | create-test |
| Coverage Gap | `payment.mapper.ts` | Data transformation logic — no spec file | HIGH | create-test |
| Unit Test | `users.service.spec.ts` | 3 tests failing — TypeError on mock | MEDIUM | update-test |
```
